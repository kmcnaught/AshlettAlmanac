# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

A single-page "day on the water" almanac (tide, weather, sun/moon, a tagged verse or short nature-prose excerpt, a wildlife note) for a kayaker who launches from **Ashlett Creek, Southampton Water** — a *drying* creek that only floats a boat near high water. It's a Father's Day gift for someone the author knows; ship it however suits — home-screen save, hosted page, etc.

**Primary, authoritative docs already in repo — read first, don't restate:**
- `HANDOFF.md` — full architecture, data shapes, verse-selection logic, scene rendering, known limitations, the P0/P1/P2 backlog. Edit this if you change the way any subsystem works.
- `scraping.md` — how the Calshot tide scraper works, the 7-day rolling-window constraint imposed by the upstream site, and the rolling-buffer cron pattern.
- `email.md` — the open ask to tidetimes.org.uk for the unabridged 2026 PDF (the bought PDF omits the Solent secondary highs).

## Layout

- `index.html` — the page (renamed from `ashlett-almanac.html` so GitHub Pages serves it at the directory URL). Inlined SunCalc 2.0.0, no build, no CDN at runtime, no npm. Fetches `tides.json`, `verses.json`, and the wildlife data layer (`data/*.json`) at startup, with inlined fallbacks for tides/verses and a static `MARSH[]`/`NUGGET` fallback for the wildlife slot.
- `tides.json` — runtime tide data, keyed by ISO date. Read by the page via `fetch('tides.json')`. Overwritten by the scraper.
- `verses.json` — runtime verse catalogue (~150 poems + short nature-prose excerpts, multi-line schema, line-level tags). Read by the page via `fetch('verses.json')`. Hand-curated; not regenerated by any script. See `HANDOFF.md §7`.
- `data/*.json` — runtime wildlife data (scraped goingbirding/iNaturalist/eBird sources + computed seasonality + per-species notes + Wikipedia descriptions + hand-curated NVC saltmarsh bedrock plants and animals). Feeds the in-page picker that surfaces one species per day in the "in the saltmarsh now" slot. See `HANDOFF.md §8` for the pipeline and refresh procedure.
- `scripts/scrape_calshot_tides.py` — fetches a date range from tidetimes.org.uk and writes `tides.json`. Usage: `python3 scripts/scrape_calshot_tides.py --start YYYY-MM-DD --end YYYY-MM-DD --out tides.json`.
- `scripts/scrape_{goingbirding,inaturalist,ebird}.py` + `compute_seasonality.py` + `build_{species_canonical,species_notes,wildlife}.py` + `fetch_species_descriptions.py` — the wildlife data pipeline. Each script has a comment-block header with args + a worked example. See `HANDOFF.md §8a` for the order they run in.
- `scripts/smoke_test_html.py` + `smoke_test_html.js` — headless Node harness that loads the page with stubbed DOM + fetch + localStorage and asserts the main sections render in both the fetched-data and fallback paths. Run after non-trivial JS changes.
- `.env` — gitignored; holds `EBIRD_API_KEY` for the eBird scraper. Re-issue from https://ebird.org/api/keygen if lost.
- `tide-times-calshot-2026.pdf` / `files.zip` — purchased reference material.

## How the pieces fit

1. On load, the almanac fetches `tides.json` (keyed by ISO date) and merges it over an inlined `__sample__` fallback. If today's date isn't in the fetched data — or the fetch fails entirely — it falls back to the sample and shows a warning.
2. It also fetches `verses.json` (the verse/prose catalogue). If the fetch fails, an inlined `VERSES_FALLBACK` of ~15 strong entries kicks in — the page still picks a fitting line for the day.
3. The verse picker scores entries against the day's signals (moon, weather, tide, season, time of day), filters by reason-priority, and picks the **least-recently-shown** entry from the matching pool (tracked in `localStorage` under `verseSeen`). Within the chosen entry, the line whose tags best overlap with today's signals wins. See `HANDOFF.md §7e`.
4. The scraper exists because the bought PDF is **abridged to 2H+2L per day**, but Southampton Water has a **double high water (4H+2L)** that matters for navigation. The scraper pulls the full 4H+2L from per-day public pages and writes `tides.json` directly.
5. Upstream gate limits the scraper to **today + ~6 days** — anything further silently returns today's data, which the scraper detects via the page's date heading and skips. So the scraper cannot populate a multi-week gift window on its own; the unabridged PDF (see `email.md`) is the path to full coverage.
6. The page also fetches live weather from Open-Meteo (no key, CORS-open). It degrades gracefully offline.
7. Sun/moon are computed offline via the inlined SunCalc.

## Editing `index.html` — load-bearing gotchas

These are easy to break and hard to spot. The full list is in `HANDOFF.md §12`; the must-knows:

- **Do not add `<text>` elements to `#curve` SVG.** It uses `preserveAspectRatio="none"` and will distort glyphs. The axis labels and legend are deliberately HTML beneath the SVG, not SVG text.
- **Paddleable window = ±1h around each HW pair**, no height threshold. The rule comes from the recipient ("for kayaks it's good an hour before first high tide to an hour after second high water"). Encoded in `paddleWindows(ex)` in `index.html`; don't bolt a height check back on without a real reason from him.
- **`localStorage` is allowed** (for poem-rotation history etc.). This breaks Claude Artifacts compatibility but home-screen-saved use on iOS/Android Safari/Chrome still works. Don't rely on it for anything safety-critical (clearing site data will wipe it); the page must still render sensibly with no stored state.
- **Tides → cosine interpolation, not a harmonic model.** The double HW only appears because the data lists both highs. Always list both highs per day when extending `tides.json`.
- **`tides.json` is fetched async at startup.** Tide-dependent rendering (curve, headline, verse first pick) lives inside an `async` IIFE near the end of the script. The verse helpers (`renderVerse`, `activeReasons`, `pickVerse`) live inside that IIFE too because they close over `range`. When opened via `file://`, some browsers (Chrome) block the fetch — the `__sample__` fallback covers that case.
- **Timezone:** display uses the viewer's local zone via `getHours()`. `CONFIG.tz` only shapes the Open-Meteo query.
- **Verse data:** the catalogue is in `verses.json` (mix of poems and short nature-prose excerpts). `HANDOFF.md §7` is the authoritative reference for the schema (multi-line entries with line-level tags), the controlled tag vocab, the URL-must-come-from-a-live-search-result rule (the *Dover Beach* Wikisource path is the cautionary tale), the prose-excerpt convention (`isProse: true`), and the localStorage rotation contract. Public-domain status no longer required — quote ~one line, link to the canonical host. **Don't rename `id` fields on shipped entries** — they're the localStorage rotation key, so renaming wipes the recipient's rotation history for that entry.

## Per-day tide-data quality flags

`tides.json` entries support two booleans:
- `verified: true` — real predictions; clears the on-page "sample data" warning.
- `abridged: true` — only principal extrema (2H+2L); surfaces a footnote that Solent secondary highs are missing. Use for any day backed by the **PDF only**. Days from the scraper (4H+2L) leave this off.

## Build / test / run

- **Build:** none. Serve `index.html` + `tides.json` + `verses.json` from the same directory (any static host, or `python3 -m http.server` locally). Opening the HTML directly via `file://` also works — both inline fallbacks kick in if the browser blocks the fetches.
- **Tests:** `python3 scripts/smoke_test_html.py` — a headless Node harness that stubs `document` + `fetch`, evals the app `<script>`, and asserts the main DOM writes. Run after non-trivial JS changes.
- **Scraper:** `python3 scripts/scrape_calshot_tides.py --start … --end … --out tides.json` (stdlib only — no virtualenv needed). Default 0.5 s politeness delay; run no more than daily.

## Hosting (GitHub Pages + Actions)

The page is wired to run on GitHub Pages, with a scheduled Action keeping `tides.json` fresh:

- `.github/workflows/scrape-tides.yml` — daily cron (06:10 UTC) + manual dispatch. Runs the scraper for today + 6 days and commits `tides.json` if it changed. Needs `permissions: contents: write` (set in the workflow).
- `.github/workflows/smoke-test.yml` — runs the headless harness on every push / PR that touches the page, data, or harness.

**One-time manual setup** (can't be scripted without admin tokens):
1. Repo → Settings → Pages → Source: *Deploy from a branch* → `main` / root.
2. Repo → Settings → Actions → General → Workflow permissions: ensure *Read and write permissions* is enabled (or rely on the per-workflow `permissions:` block, which this workflow uses).

After that, each scrape commit triggers a Pages rebuild and the live page picks up the new tide data on next load.

## Conventions

- `scripts/` follows the user's global rule (`~/.claude/CLAUDE.md`): rerunnable/tweakable scripts live here with a comment header (purpose, args, example). The existing `scrape_calshot_tides.py` is the template.
- Don't "fix" the known limitations called out in `HANDOFF.md §13 — Known limitations`; they're design choices (cosine interpolation, weather-driven verse changing intraday, wildlife lines as the paddler's to correct).
