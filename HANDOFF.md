# Ashlett Creek Almanac — Handoff Documentation

A single-page "day on the water" card for one person: tide, weather, sun, moon, a
public-domain verse, and a local wildlife note, styled as a soft watercolour scene
that recolours to the day's conditions. Built as a Father's Day one-off for a kayaker
who launches from **Ashlett Creek, Southampton Water** (a *drying* creek — it only
floats a boat near high water), plays folk tin whistle, and writes poetry.

---

## 1. Quick facts

| | |
|---|---|
| **Files to ship** | `index.html` + `tides.json` + `verses.json` (served from same directory). |
| **Build step** | None. Drop the files on any static host, or open the HTML directly — inline fallbacks cover `file://` if the browser blocks the fetches. |
| **Dependencies** | SunCalc 2.0.0 (MIT) is **inlined** — no CDN, no npm install at runtime. |
| **API keys** | None. |
| **Network** | Open-Meteo (weather) + same-origin `tides.json` + same-origin `verses.json` fetches. All three degrade gracefully (inline fallbacks). |
| **Persistence** | `localStorage` (`verseSeen`, `headlineSeen`) for least-recently-shown rotation of verses and headline variants. The page also renders correctly with no stored state. |
| **Target device** | His phone, mostly. Mobile-first, max-width 430px, also fine on desktop. |

Ship however suits — host the two files on a static site so `tides.json` can be
regenerated by a scheduled scraper, or save the HTML alone to a home screen and rely
on the inlined sample fallback. No server, router, or framework.

## 2. What's implemented

A single card, top to bottom:

1. **Sky scene + date picker** — full-bleed gradient that *is* the weather: clear day,
   overcast, mist, rain (animated streaks), or a starlit night with a moon. The place
   title sits over a slim date picker — `‹  Wed 24 Jun  ›` chevrons with a `[Today]`
   pill that appears only when off today. Range is **today + 6 days** (matches the
   scraper window). Selection persists for the same calendar day via
   `localStorage.lastViewed`; on a fresh day it defaults back to today.
2. **Headline** — one plain-language observation chosen by a 15-branch priority
   cascade (weather, tide range, dawn/sunset HW alignment, full moon, etc), plus a
   sub-line. Each branch holds a small pool of variants, rotated least-recently-shown
   via `localStorage` (`headlineSeen`), so the same condition gives a different line
   on repeat. **Only today's view burns the rotation** — previewing future days uses
   a deterministic first-option pick so navigation doesn't perturb today's sequence.
3. **Verse of the day** — a public-domain poem teaser line + poet/year + a link to the
   **full poem on its real source**. The poem is *chosen to fit the day* (weather,
   moon, tide, season) and the eyebrow says why ("A verse for a blowy day"). The verse
   is **anchored to today's signals**: navigating to a future day does **not** re-pick
   the verse.
4. **Tide at Ashlett** — an inked SVG curve with the **double high water**, a shaded
   "paddleable" band (±1h around each HW pair), the moon-up bar; for the selected day.
   The **now-marker only renders when viewing today**. Below it a **time axis**
   (12am · 6am · noon · 6pm · 12am) and a **legend**; then the HW/LW times+heights and
   a sun/moon rise-set line.
5. **In the saltmarsh now** — a month-specific wildlife/flora line for the selected
   day's month (occasionally a local-history nugget).
6. **Weather + Moon** footer row — temperature, conditions, wind; moon phase glyph,
   name, % lit. On today: live current conditions. On a forecast day: representative
   midday slice from the cached hourly forecast.

Everything except live weather is computed **offline** on load, then the headline,
scene, and verse are **refined** once weather arrives. Navigation re-runs the per-day
render against the cached weather response — there's only one weather fetch per page
load.

## 3. File anatomy

It's one HTML file with three parts in `<head>`/`<body>`:

- **`<style>`** — design tokens (CSS variables: ink, parchment, water, marsh, moon),
  then component styles. Palette is estuary-derived (water/mud/saltmarsh/mist), not a
  generic theme.
- **DOM** — static skeleton with `id`s the script fills (`#sky`, `#headline`,
  `#verse-line`, `#curve`, `#extremes`, `#light`, `#marsh`, `#wx-temp`, `#moon-name`, …).
- **First `<script>`** — the **inlined SunCalc 2.0.0** UMD bundle. It attaches
  `window.SunCalc` in the browser branch of its wrapper.
- **Second `<script>`** — all app logic (config, data, helpers, compute, render,
  weather). This is the file you'll mostly edit.

> **Source-of-truth note:** I authored this from a `template.html` with a
> `/*__SUNCALC_LIB__*/` marker and a tiny Python step that injected the SunCalc source.
> You do **not** need that workflow — `index.html` is standalone and editable
> directly. The only reason to revisit injection is upgrading SunCalc (see §12).

## 4. Config

```js
const CONFIG = {
  lat: 50.812, lon: -1.333,   // Ashlett Creek, near Fawley
  tz:  "Europe/London"         // only used for the Open-Meteo query
};
```

- The **paddleable window** rule is the recipient's: an hour before the first HW of a
  cycle to an hour after the last HW of that cycle. Encoded in `paddleWindows(ex)` —
  no height threshold, by design (the recipient knows the creek; the page just states
  the window).
- Times shown to the user render in the **viewer's local timezone** (BST in the UK).
  SunCalc returns UTC `Date`s; `getHours()` localises them. `CONFIG.tz` does **not**
  affect display — only the weather request.

## 6. Tide system

### Data shape (`tides.json`)
```json
{
  "2026-06-21": {
    "verified": true,
    "extrema": [
      { "t": "06:20", "h": 4.3, "type": "HW" },
      { "t": "08:10", "h": 3.9, "type": "LW" }
    ]
  }
}
```
List **both highs** per day to keep the Solent double HW visible. `verified: true`
clears the sample-data warning; `abridged: true` (PDF-only days, 2H+2L) surfaces a
footnote that secondary highs are missing.

### Loading
`tides.json` is fetched at startup via `loadTideTable()`, then merged over the
inlined `TIDE_FALLBACK` (`{ "__sample__": {...} }`). If today's ISO date isn't in
the merged table — or the fetch fails entirely (offline, `file://` in Chrome,
scraper down) — the page falls back to `"__sample__"` and shows the sample-data
warning. All tide-dependent rendering lives inside an `async` IIFE that awaits this
load; the verse helpers live inside that IIFE too because they close over `range`.

### Maths
- **`heightAt(hour, ex)`** — piecewise **cosine** interpolation between consecutive
  extrema. This is a smooth visual approximation, **not** a real harmonic tide model.
  The Southampton double high water appears only because the *data* lists two highs.
- **`paddleWindows(ex)`** — groups today's HW extrema into cycles (consecutive HWs
  within ~3.5h are one cycle, capturing Southampton's double HW) and returns
  `[start,end]` hour-ranges = `[first HW − 1h, last HW + 1h]` per cycle, clamped to
  `[0,24]`. No height threshold; the rule comes from the recipient.
- `range = dayHigh − dayLow` drives spring/neap logic (springs `> 4.2`, neaps `< 2.6`).

### Curve rendering (`#curve` SVG, viewBox `0 0 400 122`)
Layers, back to front: paddleable bands → area fill → tide
path → sun-up bar (top) → moon-up bar (bottom) → now vertical → now dot. The **time
axis** and **legend** are HTML beneath the SVG. A small icon column sits to the left
of the SVG (`.curve-icons`) with a static sun glyph and a phase-shaped moon glyph
(sliver / quarter / gibbous / full, rendered by `renderChartMoonIcon` from
`SunCalc.getMoonIllumination`'s `fraction` + `phase`).

The curve interpolates over a window slightly **wider** than midnight–midnight:
`exFull` prepends yesterday's last extremum (`t -= 24`) and appends tomorrow's first
(`t += 24`) from `TIDE_TABLE`, then `heightAt(h, exFull)` is sampled for `h∈[0,24]`.
Without those anchors the curve flatlines for the hours before the day's first
extremum and after the day's last. The displayed `#extremes` list still uses today's
`ex` only. **`paddleWindows` also reads `exFull`** so a HW pair that straddles
midnight (yesterday's last HW + today's first HW within ~3.5h) is grouped as one
cycle and the chart shows a single continuous band rather than two singletons with a
phantom gap. This relies on `tides.json` containing yesterday's entry — the daily
scrape job uses `--merge` and `--start "$(date -I -d '-1 day')"` for that reason.

> **Critical gotcha:** the SVG uses `preserveAspectRatio="none"` (it stretches to fill
> width). **Never put `<text>` inside it** — glyphs distort horizontally. That's why the
> axis labels and legend are HTML, not SVG.

## 7. Verse system — sourcing, tagging, validation

### 7a. The set (~150 entries — poems + short nature-prose excerpts)

The set is no longer hand-counted in this doc; the live count is whatever's in
`verses.json`. As of this writing: **~152 entries** spanning Wang Wei (8th c.) to
Kathleen Jamie (alive), with ~11 prose excerpts from J. A. Baker's *The Peregrine*
and Henry Williamson's *Tarka the Otter* — the poetry-vibed prose covers the
Solent species the canon doesn't (glasswort, sea-thrift, wigeon, redshank,
shelduck, bar-tailed godwits, eelgrass / *zostera*).

The original 9-poem PD-only set is preserved as the inline `VERSES_FALLBACK` plus
six additions (Sea-Lavender, Darkling Thrush, Spring, To Autumn, Wild Geese,
Tarka glasswort, Peregrine shelduck) so the page still renders meaningfully if
`verses.json` is missing.

**Data shape — entry:**
```js
{
  id: "longfellow-tide-rises",         // stable; used as the localStorage rotation key
  title: "The Tide Rises, the Tide Falls",
  poet: "Henry Wadsworth Longfellow",  // for prose this is the author
  year: 1882,                          // null for living poets
  url: "https://...",                  // verified canonical full-text URL
  isProse: true,                       // optional — flips render to "from <book> by <author>"
  source: "The Peregrine, 1967",       // optional — only on prose entries
  lines: [
    {text: "...", tags: ["tide", "water"]},     // line-level tags drive which line shows
    {text: "...", tags: ["evening", "wader"]},
    ...
  ],
  tags: ["tide", "sea", "evening", "wader", ...]  // entry-level — union of all line tags;
                                                  // used for the pool filter
}
```

### 7b. How they were SOURCED
The original 9 were public-domain only, hand-fetched. The expansion was done with
**parallel research agents** themed by gap (tide / moon / weather / coast /
season / wildlife / women & contemporary / shorebirds / saltmarsh flora /
working coast / translated voices / nature-prose). See git history of this doc
for the agent briefs.

Key sourcing rules that carried through:
- **The URL must come from a live web search result, not constructed.** The
  *Dover Beach* gotcha (lives at `/wiki/The_poetical_works_of_Matthew_Arnold/Dover_Beach`
  — a collection subpage — not `/wiki/Dover_Beach`) is why. Guessing fails.
- **Web-fetch the URL and verify each quoted line letter-for-letter** against the
  page text. Period spellings preserved ("to-night" not "tonight").
- **Public domain is NOT required.** Quote ~one line / one sentence, link to the
  host. Copyrighted poems on reputable hosts (Poetry Foundation, poets.org,
  Wikisource, Scottish Poetry Library, Poetry Archive, Modern Poetry in
  Translation, the Onassis Cavafy archive, etc.) are welcome.
- **Prose excerpts** must NAME the gap species directly (glasswort, shelduck,
  zostera, etc.) and read with a "poetic vibe". One sentence preferred,
  max two.

Hosts that 403 most agent fetches: Poetry Foundation, Granta, Bloodaxe, Poetry
Society UK. Where the canonical URL is on one of these, we used it anyway if the
text was verified through another mirror.

### 7c. Tag vocabulary (controlled — extend with care)

```
Season:     spring, summer, autumn, winter
Time:       dawn, evening, night
Sky/wx:     sun, rain, mist, wind, sky
Water:      sea, tide, water, river, marsh, creek
Moon:       moon
Wildlife:   bird, geese, wader, fish, flora
Register:   folk (mythic/ballad), quiet (stillness), migration (movement/return)
```

Adding a tag means **also** updating `activeReasons` and/or `SEASON_TAG` to use
it — orphan tags don't affect picking.

Tag at two levels:
- **Entry-level `tags`** — union of all line tags plus any thematic extras. Used
  for the first-pass pool filter.
- **Line-level `tags`** — used to pick *which line* of a multi-line entry to
  show. A poem with both a "high tide" line and a "winter" line will show
  whichever scores higher against today's combined signal set (see §7e).

### 7d. How entries were VALIDATED
1. **Link resolves** — URL from a live search result, page contains the quoted
   text (or, where 403'd, cross-checked via another reputable mirror).
2. **Teaser-line accuracy** — checked letter-for-letter against the page.
3. **Tag accuracy** — line tags reflect the line itself, not the poem's theme.
4. **Coverage** — every `activeReasons` tag and every `SEASON_TAG` value has
   ≥ 4 matching entries. Verified by:
   ```bash
   python3 -c "import json, collections; t=collections.Counter();
   [t.update(e['tags']) for e in json.load(open('verses.json'))]; print(t)"
   ```

### 7e. How the day's verse is CHOSEN

**Two-stage:** pick an *entry*, then pick the *line* within that entry.

**Stage 1 — entry.** `pickVerse(weather)` walks `activeReasons` in priority
order. For the first reason whose tag has ≥ 1 matching entry, it picks the
**least-recently-shown** entry from that pool (deterministic tiebreak by `id`).
Last-shown is tracked in `localStorage` under `verseSeen` keyed by entry id;
value is `year*1000 + dayOfYear`. The picker still renders correctly with no
stored state (first-load behaviour).

**Stage 2 — line.** Within the chosen entry, `pickLine(v, signals)` scores each
line by how many of *today's* signals (moon + weather + tide + season + time of
day) overlap with the line's tags. Highest score wins. This is what lets a
multi-line poem show its "high tide" line on spring days and its "winter" line
on cold days — the user's "saltmarsh + winter + high tide" vs "saltmarsh +
winter + sunny" distinction.

| Priority | Condition | Eyebrow label | Tag |
|---|---|---|---|
| 1 | `nearFull` (`|phase−0.5| < 0.06`) | for the full moon | `moon` |
| 2 | new moon (`phase < .04 || > .96`) | for the new-moon dark | `night` |
| 3 | weather `kind==="rain"` | for the rain | `rain` |
| 4 | weather `wind ≥ 22` km/h | for a blowy day | `wind` |
| 5 | weather `kind==="fog"` | for the mist | `mist` |
| 6 | tide `range > 4.2` (spring) | for the spring tide | `tide` |
| 7 | tide `range < 2.6` (neap) | for the quiet neap | `water` |
| 8 | clear & `wind < 14` | for a bright, still day | `sun` |
| 9 | fallback | for {Month} | `SEASON_TAG[month]` |

```js
// month 0–11 → preferred tag when nothing stronger fires
const SEASON_TAG = ["winter","winter","spring","spring","spring","summer",
                    "summer","summer","autumn","autumn","autumn","winter"];
```

`pickVerse` is called twice: first synchronously after VERSES + TIDE_TABLE load
(with `weather = null`) so something renders immediately; then again from
`loadWeather()` once Open-Meteo replies, refining the pick. Both calls update
`verseSeen`; the rotation forgives the double-mark since it spans the whole
catalogue.

**Eyebrow:** "A verse {label}" for poems; "A passage {label}" for prose. The
"by" line shows poet (year) + read-the-full-poem link for poems, or
"from *Book* by Author + read more" for prose.

### 7f. To add a poem / prose excerpt
- Append `{id, title, poet, year, url, lines:[{text,tags}], tags}` to
  `verses.json`. Add `isProse:true` and `source:"Book Title, Year"` for prose.
- The `id` is the localStorage rotation key — once shipped, **don't rename
  ids** or recipients lose their rotation history for that entry.
- Tags must come from the controlled vocab in §7c.
- `verses.json` should remain valid JSON; if you edit by hand, validate with
  `python3 -c "import json; json.load(open('verses.json'))"`.

To re-map a condition: edit `activeReasons` in `index.html`. To change
the seasonal bias: edit `SEASON_TAG`. To change the picker logic: it's all in
one async IIFE in the script, around `function pickVerse(weather)`.

## 8. Wildlife / flora system

The wildlife slot is now a **runtime data-driven picker** (since v2). It reads six
JSON files at boot and surfaces one species per day with a contextual eyebrow,
species name + link, lead line, and optional observer attribution. The static
`MARSH[]` array and `NUGGET` constant are retained as the **offline / `file://`
fallback** — if any critical data file fails to fetch, the page falls back to the
old behaviour (`MARSH[dayMonth]` with `NUGGET` rotating every 9th day-of-year).

### 8a. Data pipeline (offline, hand-runnable — see scripts/)

```
            ┌── scrape_goingbirding.py  ─── data/goingbirding_birds.json
            │   (HOS sightings @ Ashlett/Calshot/Fawley, 5 sites × 1 year)
            ├── scrape_inaturalist.py   ─── data/inaturalist_obs.json
            │   (research-grade, 5 km west-shore box, 1 year)
            ├── scrape_ebird.py         ─── data/ebird_hotspots.json
            │   (3 hotspot species lists, EBIRD_API_KEY from .env)
            │
            ├── build_species_canonical.py ── data/species_canonical.json
            │   (manual goingbirding↔eBird name join; feral + exclude flags)
            │
            ├── compute_seasonality.py  ─── data/seasonality.json
            │   (wrapped-Gaussian density per species, peakiness,
            │    localRarity within iconicTaxon, joins gb+iNat by sci name)
            │
            ├── build_species_notes.py  ─── data/species_notes.json
            │   (layered auto-observer/manual/bimodal/seasonality notes;
            │    reads observer_notes_overrides.json for FP excludes)
            │
            └── fetch_species_descriptions.py ── data/species_descriptions.json
                (iNat /v1/taxa for Wikipedia summary + default photo URL)

Hand-curated, no scripts:
    data/saltmarsh_bedrock.json          (16 NVC saltmarsh plants)
    data/saltmarsh_bedrock_animals.json  (14 saltmarsh / mudflat animals)
    data/inaturalist_excludes.json       (escapee taxonIds — Holbury grackle)
    data/observer_notes_overrides.json   (regex false-positive excludes)
```

**Total data layer**: ~1 MB of JSON. ~50 HTTP requests to refresh the scraped
sources; the bedrock files don't move.

### 8b. Refreshing in a year

Re-running everything from scratch:

```sh
# Scrape — ~3 min total wall time at default 1 s delays
python3 scripts/scrape_goingbirding.py --chunk-days 45
python3 scripts/scrape_inaturalist.py
python3 scripts/scrape_ebird.py   # needs EBIRD_API_KEY in .env

# Build (depends on scrapes)
python3 scripts/build_species_canonical.py
python3 scripts/compute_seasonality.py
python3 scripts/build_species_notes.py
python3 scripts/fetch_species_descriptions.py  # ~6 min, iNat rate limit
```

Each script has a comment-block header documenting its args + a worked example.
Order matters — canonical → seasonality → notes — but every script can be
re-run in isolation if you change one thing (e.g. add a manual note, then just
re-run `build_species_notes.py`).

### 8c. Things that may break in a year

- **Goingbirding HTML** (`scrape_goingbirding.py`): the parser is a regex on the
  page's `<tr>…</tr>` structure. ASP.NET sites tend to be stable but if the
  markup changes the regex breaks silently (you'd get 0 records). Verify with
  the smoke check at the top of the script's output. The Hampshire OS site is
  unofficial — there is no API or rate-limit doc.
- **iNaturalist `/v1/taxa`**: the `path-based` multi-ID endpoint
  (`/v1/taxa/{id1,id2,...}`) is what gives us `wikipedia_summary` —
  the query-based form (`/v1/taxa?taxon_id=…`) returns an abbreviated record.
  If iNat tightens this, fall back to `/v1/taxa/{id}` per species (slower).
- **eBird API key**: stored in `.env` as `EBIRD_API_KEY=…` (gitignored). Free,
  but tokens can be revoked — re-request at https://ebird.org/api/keygen.
- **iNat escapees**: the Holbury garden Great-tailed Grackle is the known
  escapee (taxonId 9607). If a new feral species shows up in the records,
  add it to `data/inaturalist_excludes.json`.
- **Goingbirding observer changes**: J G Ross is the main Ashlett patcher; his
  quotes are most of the auto-observer notes. If he stops logging, the picker
  will lean more on Wikipedia descriptions and bedrock notes.

### 8d. Picker logic (in `index.html`, inside the async IIFE)

- Pool: records-based stable species (n≥3 with at least one note) + bedrock
  plants + bedrock animals. Feral and excluded species dropped.
- Score = `season_match × rarity_sweet × usage_decay`:
  - **season_match**: wrapped-Gaussian density at this week (records-based) or
    month-membership (bedrock).
  - **rarity_sweet**: bell over `localRarity` peaked at 0.4 (avoids the
    always-here commons AND the once-off vagrants).
  - **usage_decay**: `0.05 + 0.95·(1 − e^(−Δ/halflife))` from
    `localStorage.wildlifeSeen`. Halflife = 21 days.
- **Group balance**: weighted-random group pick by
  `{birds: 0.55, plants: 0.20, insects: 0.10, others: 0.15}` first, then
  weighted-random species inside the chosen group.
- **Determinism**: mulberry32 PRNG seeded by `hash(ISO date)`. Same date =
  same pick on refresh, until `wildlifeSeen` shifts.
- **Framing → eyebrow**: per species type, "Passing through around now" for
  peaky migrants, "Listen for at the creek now" for warblers, "In the
  saltmarsh now" for mid-marsh plants in flower, "On the mud now" for lugworm
  / mud snail etc.
- **Lead line priority**: auto-observer quote > manual note > bimodal note >
  Wikipedia summary > internal seasonality category text.
- **NUGGET rotation** (every 9th day-of-year) still takes over the slot from
  the picker — this preserves the original local-history beat.

For an offline preview of what the picker would show over N days, run
`python3 scripts/build_wildlife.py --days 90 --print`. The script mirrors
the JS picker's logic and writes a dated JSON to `/tmp` by default + prints
a UI-styled preview to stdout (eyebrow / species / lead line / caption /
link per day). Nothing in the live page consumes the JSON.

### 8e. Tuning knobs

In `index.html`:
- `WILDLIFE_TARGET` — change the per-group share (birds-heavier in winter
  when invert pool collapses?).
- `WILDLIFE_DECAY_HALFLIFE` — bump up to space repeats further.
- The eyebrow phrase tables in `makeEyebrowWildlife()`.
- The species link priority in `renderMarshSlot()`.

In the data layer:
- `MANUAL_NOTES` in `scripts/build_species_notes.py` — hand-added context per
  species (re-run the script after editing).
- `data/observer_notes_overrides.json` — kill regex false positives without
  touching the script.
- `data/saltmarsh_bedrock*.json` — add/remove species; flowering and visible
  windows are months 1-12 arrays.

## 9. Headline engine (`buildHeadline`)

15-branch priority cascade, first match wins. Each branch has a pool of 2-3 variants;
within the matching branch, `pickFrom(branchId, pool)` returns the least-recently-shown
variant and records the pick under `localStorage["headlineSeen"]` keyed by
`${branchId}:${poolIndex}` (value = `year*1000 + dayOfYear`, same shape as `verseSeen`).
First-load picks deterministically (all `seen` lookups are `undefined`, sort is stable).

| # | Branch id | Condition | Pool size |
|---|---|---|---|
| 1 | `fullmoon` | `nearFull` & clear/cloud & evening HW (17.5–22.5h) & moon up at HW | 3 |
| 2 | `heavyrain` | `kind==="rain"` & `precipToday ≥ 8 mm` | 3 |
| 3 | `rain` | `kind==="rain"` (any) | 3 |
| 4 | `fog` | `kind==="fog"` | 3 |
| 5 | `wind` | `wind ≥ 22 km/h` | 3 |
| 6 | `king` | `range > 4.7 m` | 2 |
| 7 | `lastdry` | `dryToday && wetAhead` | 2 |
| 8 | `spring` | `range > 4.2 m` | 3 |
| 9 | `dawnhw` | HW within `[sunriseH−0.5, sunriseH+2]` & clear/cloud & `wind < 18` | 2 |
| 10 | `sunsethw` | HW within `[sunsetH−1.5, sunsetH+0.5]` & clear/cloud & `wind < 18` | 2 |
| 11 | `nightonly` | paddle windows exist but none overlap daylight (`sunriseH..sunsetH`) | 2 |
| 12 | `low` | no paddle windows today (`windows.length === 0`; defensive — only fires if today's data has no HW extrema) | 2 |
| 13 | `neap` | `range < 2.6 m` | 3 |
| 14 | `glassy` | not rain & `wind < 14` & a daylit window exists | 3 |
| 15 | `default` | fallback (windowText, or "Today on the water.") | 1 |

Weather inputs (computed from Open-Meteo `daily.precipitation_sum`):
`dryToday = precip[0] < 1mm`; `wetAhead = precip[1] ≥ 2 && precip[2] ≥ 2`;
`precipToday = precip[0]`. Offline fallback (`loadWeather` catch) passes
`{wind:8, kind:"cloud", precipToday:0, dryToday:true, wetAhead:false}`.

**Copy conventions** (don't violate when adding pools): factual/observational voice.
No second-person directives ("you can X"), no instructional folksiness ("pub boots
advised", "if you're going, go now"), no explanations of what tide conditions *mean* —
the recipient is an experienced kayaker on Ashlett. State numbers and named features;
let him judge. The historical example of the wrong voice is in git: the original
7-branch cascade ("A big spring tide — the marsh fills right up; you can glide over
ground that's usually grass.") was rewritten because it was patronising.

Also:
- **Spell out "high water" / "low water"** in user-facing text. HW/LW are fine in
  code, comments, and the `#extremes` row (which already renders "High"/"Low").
- **Don't treat HW as a single point.** Southampton Water has a double high water,
  so the data lists multiple highs per day. Branches that align to a time-of-day
  alignment (`dawnhw`, `sunsethw`, `fullmoon`) `filter` rather than `find`, then
  join times with `" and "` — so the headline says "high water at 5:30am and 7:15am"
  on a true double-HW day, but degrades to "high water at 5:30am" if only one falls
  in the window. Branches that talk about tide *size* (`spring`, `king`, `neap`)
  reference `dayHigh`/`dayLow`/`range`, not specific HW times.
- **Prefer time-of-day phrasing** ("at first light", "as the sun gets low", "at
  dusk", "in the evening") when the branch itself is defined by alignment to that
  time. The clock time is still listed as supporting detail; the phrase is the
  hook.

## 10. Scene rendering (`scene(kind, isDay)`)

`weather_code` → `WMO` map → `kind` ∈ `clear | cloud | fog | rain` (+ `isDay`). Each sets
a gradient and toggles elements (sun, moon, stars, mist, rain). Night always shows stars;
a clear night shows the moon orb.

---

## 11. Graceful degradation

If Open-Meteo fails, the `catch` in `loadWeather` paints a neutral cloud scene, shows
"weather offline", still renders tide + sun + moon, and runs `buildHeadline` with safe
defaults plus a small notice. The page is never blank.

## 12. Editing & gotchas for Claude Code

- **Edit `index.html` directly.** No build to ship.
- **`tides.json` is fetched async.** The tide-and-below block is wrapped in an
  `async` IIFE that awaits `loadTideTable()` + `loadVerses()` + `loadWeather()`.
  Per-day rendering lives in `renderDay(iso)` inside that IIFE; the boot path calls
  it twice (offline, then after weather arrives) and the chevron handlers call
  `go(iso)` which re-renders without re-fetching. If you add new tide-derived
  values used per-day, compute them **inside `renderDay`** so they refresh on
  navigation.
- **Verse stays today.** `pickVerse()` runs against TODAY's signals
  (`todayRange`, `todayPhase`, `todayMonth`, etc.) and the verse is **not**
  re-picked when the user navigates dates. Don't re-introduce a verse rerender
  inside `renderDay`.
- **Headline rotation only burns on today.** `buildHeadline()` takes the selected
  `iso`; `pickFrom` short-circuits to `pool[0]` when off today so previewing future
  days doesn't perturb the `headlineSeen` localStorage.
- **Now-marker is today-only.** The "now" dot+line on `#curve` only renders when
  `iso === todayISO`. Don't draw it for previewed future days.
- **Date-picker contract.** `selectedISO` is module-level; `go(iso)` clamps to
  `[todayISO, NAV_BOUND_LAST]`, writes `lastViewed`, then `renderDay(iso)`.
  `__almanac.go(iso)` is a test seam (used by the smoke harness) — keep it.
- **Don't add `<text>` to `#curve`** — `preserveAspectRatio="none"` distorts it. Axis &
  legend live in HTML for this reason.
- **`localStorage` is in use** for poem-rotation history (least-recently-shown picker).
  This breaks Claude Artifacts compatibility — gone deliberately. Home-screen-saved use
  on iOS/Android browsers still works. The page must still render sensibly with no
  stored state (first visit, or after the user clears site data).
- **Timezone:** display is the viewer's local zone via `getHours()`; `CONFIG.tz` only
  shapes the weather query.
- **Upgrading SunCalc** (rarely needed): download
  `https://registry.npmjs.org/suncalc/-/suncalc-2.0.0.tgz`, extract `package/suncalc.cjs`,
  replace the first `<script>` body with it. Its UMD wrapper sets `window.SunCalc` in the
  browser. (npm/registry is reachable; raw.githubusercontent paths for SunCalc 404'd —
  use the tarball.)
- **`dayIdx`** = day-of-year; drives verse rotation within a tag pool and the wildlife
  nugget cadence.

### Testing approach already in use
A headless Node harness: load the inlined SunCalc via `require`, stub
`document.getElementById` and `fetch`, `eval` the app `<script>`, then assert the
synchronous DOM writes (verse line + verified URL, tide-window sentence, light line,
curve paths) and both headline branches (offline and a simulated weather day). Re-run
after every change. Recommended to keep this as the regression check.
